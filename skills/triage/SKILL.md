Read `${CLAUDE_PLUGIN_ROOT}/context/workflow.md` to understand the full bm-kit pipeline and where this skill fits.

Move `.scratch/` issues through a state machine by reading and writing the `status` field in each issue's YAML frontmatter. No GitHub API calls — fully local.

Every comment appended to an issue during triage **must** start with:

```
> *This was generated by AI during triage.*
```

---

## Roles

Two **category** values (stored as `category` in frontmatter):

- `bug` — something is broken
- `enhancement` — new feature or improvement

Six **status** values:

- `needs-triage` — not yet reviewed
- `needs-info` — waiting on more information
- `ready-for-agent` — fully specified, safe for the ralph loop
- `ready-for-human` — requires human judgment or access
- `wontfix` — will not be actioned
- `done` — implementation complete

State transitions: `needs-triage` → `needs-info` | `ready-for-agent` | `ready-for-human` | `wontfix`. `needs-info` returns to `needs-triage` once answered.

---

## Invocation

The user invokes `/bm-kit:triage` in natural language. Interpret and act. Examples:

- "Triage the bm-kit issues"
- "Let's look at `.scratch/bm-kit/issues/06-to-prd-skill-rewrite.md`"
- "Move issue 06 to ready-for-agent"
- "What's waiting for triage in the bm-kit feature?"

---

## Show what needs attention

If no specific issue is referenced:

1. Find all `.scratch/` issue files with `status: needs-triage` or `status: needs-info`.
2. Present them grouped by feature, oldest first (by filename order within each feature).
3. Show: filename, current status, title. Let the user pick one.

---

## Triage a specific issue

1. **Read the issue.** Parse frontmatter (status, category, priority), body, and any existing `## Comments` section.

2. **Explore the codebase** if the issue touches code — use the project domain glossary (`CONTEXT.md` if present) and ADRs.

3. **Recommend.** State your recommended `status` (and `category` if missing) with brief reasoning. For bugs, trace the code path and attempt reproduction. Wait for the user's direction.

4. **Grill if needed.** If the issue is underspecified, ask targeted questions before recommending `ready-for-agent`. Do not mark an issue `ready-for-agent` without a clear scope.

5. **Apply the outcome:**
   - **`ready-for-agent`** — write an agent brief in `## Comments` explaining what to implement.
   - **`ready-for-human`** — same structure, note why it can't be delegated.
   - **`needs-info`** — append a `## Comments` entry with the needs-info template below.
   - **`wontfix`** — append a brief explanation to `## Comments`.
   - **`needs-triage`** — update status, optional comment.

   In all cases:
   - Edit the `status:` line in the frontmatter using the Edit tool.
   - Append a new entry under `## Comments` at the bottom of the file with triage reasoning.

## Quick state override

If the user says "move issue 06 to ready-for-agent", do it directly — update frontmatter, optionally ask if they want an agent brief, then act. No grilling required.

## Needs-info template

Append under `## Comments`:

```markdown
> *This was generated by AI during triage.*

**What we've established so far:**

- point 1

**What we still need:**

- question 1
```

---

## After triaging

Check all issues in the feature directory. If all issues are `ready-for-agent` (or `done`/`wontfix`), end with:

> All issues in `.scratch/<feature-slug>/` are ready. Run `/bm-kit:ralph <feature-slug>` to start the implementation loop.

Otherwise, list which issues are still `needs-triage` or `needs-info`.
